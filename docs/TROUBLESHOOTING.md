# Troubleshooting

This document collects common problems and fixes for the `nextcloud-pihole-selfhosted` stack.  
It focuses on DNS, reverse proxy, and container issues you are likely to encounter in a home lab.

---

## 1. DNS problems

### 1.1 `your.cloud.name` does not resolve

**Symptoms**

- `curl https://your.cloud.name` or a browser shows “could not resolve host”.
- `ping your.cloud.name` fails.
- `dig your.cloud.name +short` returns nothing.

**Checks**

```bash
# On the client or jumpbox
dig your.cloud.name +short
ping -c 1 your.cloud.name
getent hosts your.cloud.name
```

**Fix**

1. Ensure the client is using **Pi-hole IP** as DNS:
   - On Linux: check `/etc/resolv.conf` or `resolvectl status`.
   - On your router: set DNS to **Pi-hole IP**.
2. In the Pi-hole admin UI:
   - Go to *Local DNS* → *DNS Records*.
   - Add or confirm an entry:  
     `your.cloud.name -> Host IP`
3. Flush local DNS cache if needed:
   - Linux with `systemd-resolved`:
     ```bash
     sudo resolvectl flush-caches
     ```
   - Or simply disconnect/reconnect the network or reboot the client.

---

### 1.2 Pi-hole resolves correctly, but the client still does not

**Symptoms**

- `dig @Pi-hole-IP your.cloud.name +short` works.
- `dig your.cloud.name +short` on the same client fails.
- Browsers can’t reach `your.cloud.name`.

**Checks**

```bash
# On the client
cat /etc/resolv.conf
```

**Fix**

- Update the client’s DNS settings:
  - Replace any ISP or router DNS addresses with **Pi-hole IP**.
- If you have multiple DNS servers configured, remove the others so all queries go through Pi-hole.

---

### 1.3 `.local` hostnames behave weirdly

**Symptoms**

- `.local` names sometimes work, sometimes don’t.
- `ping delcloud.local` or similar is unreliable.
- Other names (like `.lan` or `.home`) work fine.

**Cause**

`.local` is reserved for mDNS on many systems, so DNS lookups for `.local` can bypass Pi-hole or behave inconsistently.

**Fix**

- Prefer a different suffix such as:
  - `your.cloud.name` under `.lan`, `.home`, `.internal`, etc.
- Update:
  - Pi-hole local DNS record.
  - Caddy and Nextcloud configs to use the new name.

## 2. Caddy / reverse proxy issues

### 2.1 Caddy container keeps restarting

**Symptoms**

- `docker ps` shows Caddy with `Restarting (1)` state.
- `docker logs caddy` shows config or parse errors.

**Checks**

```bash
docker ps | grep caddy
docker logs --tail 50 caddy
```

**Common causes**

- Invalid `Caddyfile` syntax.
- Conflicting settings in `compose.yaml` (e.g. `network_mode: host` plus `ports:`).

**Fix**

1. **Validate the Caddyfile:**

   ```bash
   docker run --rm \
     -v "$PWD/nextcloud/reverse-proxy/Caddyfile:/etc/caddy/Caddyfile" \
     caddy:alpine \
     caddy validate --config /etc/caddy/Caddyfile
   ```

   Fix any reported errors.

2. **Simplify the Caddy service in `nextcloud/reverse-proxy/compose.yaml`:**

   - Use either:
     - `network_mode: host` **without** `ports:`, or
     - Bridge mode **with** `ports: ["80:80", "443:443"]`.
   - Do not mix both.

3. Restart Caddy:

   ```bash
   cd nextcloud/reverse-proxy
   docker compose down
   docker compose up -d
   docker logs -f caddy
   ```

---

### 2.2 Browser warns about self-signed or untrusted certificate

**Symptoms**

- Browser shows a certificate warning.
- `curl https://your.cloud.name` fails with `SSL certificate problem`.

**Fix options**

- For quick testing with curl:
  ```bash
  curl -kI https://your.cloud.name
  ```
- For browsers:
  - Trust the internal CA certificate generated by Caddy on your client machines, **or**
  - Switch to a public domain and use Caddy with Let’s Encrypt (requires proper DNS and internet reachability).

---

### 2.3 `your.cloud.name` works by IP but not via hostname

**Symptoms**

- `https://Host-IP` in a browser shows the site.
- `https://your.cloud.name` fails.

**Checks**

```bash
dig your.cloud.name +short
getent hosts your.cloud.name
```

**Fix**

1. Ensure Pi-hole has the correct DNS record (`your.cloud.name -> Host IP`).
2. Ensure Caddy is configured with the same hostname in the Caddyfile.
3. Check that clients use Pi-hole as DNS (see section 1).

## 3. Pi-hole issues

### 3.1 Pi-hole admin UI not reachable

**Symptoms**

- `http://Pi-hole-IP:8081/admin` or `https://Pi-hole-IP:8444/admin` fails.
- DNS may still work, or not.

**Checks**

```bash
cd pihole
docker compose ps
docker logs --tail 50 pihole
ss -ltnp | grep -E '8081|8444|53'
```

**Fix**

1. Ensure the Pi-hole container is running:

   ```bash
   cd pihole
   docker compose up -d
   ```

2. Confirm the `ports:` mapping in `pihole/compose.yaml` matches what you expect (e.g. `8081:80`, `8444:443`).
3. Check host firewall rules if ports are blocked.

---

### 3.2 Docker pulls fail when Pi-hole is down

**Symptoms**

- `docker pull` or `docker compose up` returns DNS errors.
- Host uses Pi-hole as its own DNS, and Pi-hole container is not running.

**Fix**

- Configure the Docker daemon to use external DNS servers (e.g. Cloudflare, Google) so Docker does not depend on the Pi-hole container:
  - Create or edit `/etc/docker/daemon.json` on the host:
    ```json
    {
      "dns": ["1.1.1.1", "8.8.8.8"]
    }
    ```
  - Restart Docker:
    ```bash
    sudo systemctl restart docker
    ```

Pi-hole can still be the DNS for LAN clients; Docker itself can use upstream resolvers.

## 4. Nextcloud AIO issues

### 4.1 Nextcloud not reachable through Caddy

**Symptoms**

- `https://your.cloud.name` returns 502, 404, or connection refused.
- Nextcloud containers are running.

**Checks**

```bash
docker ps | grep nextcloud
ss -ltnp | grep 11000
```

Verify that the Nextcloud Apache container is listening on `127.0.0.1:11000` (or the port you configured).

**Fix**

1. Confirm the upstream in the Caddyfile matches the actual port:
   ```caddy
   handle {
       reverse_proxy 127.0.0.1:11000
   }
   ```
2. Restart Caddy:
   ```bash
   cd nextcloud/reverse-proxy
   docker compose restart caddy
   ```

---

### 4.2 Talk HPB or Collabora not working

**Symptoms**

- Nextcloud main UI is fine, but:
  - Talk calls or notifications fail.
  - Office documents do not open in the editor.

**Checks**

```bash
docker ps | grep talk
docker ps | grep collabora
ss -ltnp | grep -E '3479|9980'
```

**Fix**

1. Ensure the containers are running and ports are listening.
2. Check the Caddyfile routes:

   ```caddy
   handle_path /standalone-signaling/* {
       reverse_proxy 127.0.0.1:3479 {
           flush_interval -1
       }
   }

   handle /cool/* {
       reverse_proxy 127.0.0.1:9980
   }
   ```

3. Restart Caddy and retest from the browser.

## 5. General Docker & host diagnostics

Useful commands when “something is wrong” but you’re not sure what:

```bash
# Show running containers and their status
docker ps

# Check logs for a specific container
docker logs --tail 50 <container-name>

# Check which processes are listening on important ports
ss -ltnp | grep -E '53|80|443|8081|8444|11000|3479|9980'

# Check disk space
df -h

# Check memory and load
free -h
uptime
```

Keep this file updated as you run into new issues and solve them. Over time it becomes your personal “runbook” for this stack.
